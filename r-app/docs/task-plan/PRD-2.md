# PRD（第二轮）：监控明细、用量统计与 Token 单位优化

> 状态：ready-for-agent
> 来源：基于 `需求 2.txt` 与代码现状（第一轮 WP1–WP4 已完成）综合而成
> 适用范围：tauri-gateway（r-app，React 19 + Tauri/Rust + SQLite）
> 关联：第一轮 [PRD.md](./PRD.md) · [TASKS.md](./TASKS.md) · [progress.csv](./progress.csv)

## Problem Statement

第一轮交付了统计页/仪表盘的端点统计、实时监控、用量统计与缓存 Token 维度，但用户在实际使用中发现一批体验与正确性问题：

1. **实时请求监控的"入站/出站"列看不出端点路由。** 当前入站列只显示协议词（`claude`/`openai`），出站列只显示上游 base URL（如 `https://api.anthropic.com`）。用户真正想看的是**端点路由路径**——入站 Claude 是 `/v1/messages`、入站 OpenAI 是 `/v1/chat/completions`，出站是实际转发到上游的路径。路径才是定位请求的关键信息。
2. **首次打开"用量统计"会卡死主进程。** 进入用量统计 Tab 会触发一次全量同步，首次因为要递归扫描并解析本机大量 Claude Code / Codex 会话 JSONL，整个界面（主进程）卡死无响应。
3. **用量统计没有时间筛选。** 用户只能看到全量汇总，无法按时间段（尤其"当天"）查看，默认也不是当天。
4. **Token 大数字不可读。** 各处直接展示 `1.toLocaleString()` 后的原始数字（如 `125,000,000`），缺少"约等于多少亿/万"的辅助单位说明；而这种换算多处都需要，重复实现易不一致。
5. **仪表盘"Token（今日）"同样缺少单位辅助说明。**
6. **统计页"端点请求记录"一直处于加载中、无数据。** 统计页下半的实时监控（ranged 模式）持续显示"加载中…"，从不出数据；而仪表盘的实时监控（live 模式）正常。
7. **历史记录弹窗太小。** 历史记录弹窗布局尺寸偏小，长表格展示局促。

## Solution

围绕"监控明细的可读性与正确性"和"用量统计的可用性"做一轮针对性修复与增强，最大限度复用第一轮已建立的数据路径与组件：

1. **监控路由展示（R1）**：请求明细新增"真实入站路径 / 真实出站路径"两项数据，由代理转发汇聚点透传记录；前端监控表的"入站/出站"列改为展示路由路径（入站 `/v1/messages`、`/v1/chat/completions` 等；出站为实际转发上游路径），并保留协议/上游主机作为辅助信息（悬停或副标题）。
2. **用量同步异步化（R2）**：把 `sync_session_usage` 由同步命令改为**异步命令并在阻塞线程池执行**（`spawn_blocking`），使重量级文件扫描/解析不再占用主线程；前端进入面板时以非阻塞方式触发同步，加载期间界面可交互、其余查询照常渲染。
3. **用量时间筛选（R3）**：用量统计页新增时间段筛选器，**默认"当天"**；通过后端已支持的 `start/end`（`YYYY-MM-DD`）参数过滤总量卡片、按模型表、按天表。
4. **Token 单位工具函数（R4）**：封装一个通用格式化函数，把 Token 数量按就近量级折算为带"约等号 + 两位小数 + 中文单位"的辅助文案（如 `≈1.25亿`、`≈2.00万`、`≈900.00万`、`≈1000.00万`），多处统一复用。
5. **仪表盘 Token 单位（R5）**：仪表盘"Token（今日）"卡片用上述工具函数补充辅助小字单位说明。
6. **修复 ranged 监控一直加载（R6）**：根因是时间段计算把 `Date.now()` 直接放进 React Query 的 queryKey，使 key 每次渲染都变、查询无限重建而永远处于 pending。修复为**稳定化时间段计算**（按天对齐、记忆化），使同一筛选条件下 queryKey 稳定。
7. **历史记录弹窗放大（R7）**：增大历史记录弹窗的宽度与高度，并为表格区域提供滚动，长列表展示更从容。

## User Stories

1. 作为网关使用者，我希望实时监控的"入站"列展示客户端实际调用的路由路径（如 Claude 入站 `/v1/messages`、OpenAI 入站 `/v1/chat/completions`），以便一眼看出这条请求走的是哪种入站协议端点。
2. 作为使用者，我希望"出站"列展示请求实际转发到上游的路由路径，以便确认网关把请求发往了哪个上游端点。
3. 作为使用者，我希望即使发生了 Claude→OpenAI 的格式转换，出站路径也能正确反映真正发出的上游路径（转换后为 `/v1/chat/completions`），以便排查转换链路。
4. 作为使用者，我希望路由路径之外仍能看到上游主机/协议等辅助信息（如悬停查看完整上游 URL），以便需要时获取完整上下文。
5. 作为使用者，我希望对历史（迁移前已落库）的请求明细，路径列在无真实路径时能以协议推断值兜底展示，避免空白。
6. 作为 Claude Code / Codex 用户，我希望首次打开"用量统计"时界面不卡死，可以继续操作，以便在后台同步本机会话日志的同时浏览其它内容。
7. 作为使用者，我希望用量同步在后台线程执行、完成后自动刷新数据，以便大体量日志也不影响主界面响应。
8. 作为使用者，我希望用量同步进行中有明确的加载/进行态提示，以便知道数据正在更新。
9. 作为使用者，我希望"用量统计"提供时间段筛选器，以便只看某段时间的用量。
10. 作为使用者，我希望用量统计的时间筛选**默认是当天**，以便打开即看到今天的用量。
11. 作为使用者，我希望切换时间段后总量卡片、按模型表、按天表都随之过滤，以便不同维度口径一致。
12. 作为使用者，我希望提供常用时间段快捷项（今日/近 7 天/近 30 天/全部），以便快速切换。
13. 作为关注规模的使用者，我希望较大的 Token 数量旁有"约等于多少亿/万"的辅助小字（约等号 + 两位小数 + 中文单位），以便快速感知量级而不用数位数。
14. 作为使用者，我希望单位按数值就近的量级自动选择（接近亿用"亿"，接近万用"万"），如 `1.25亿`、`2.00万`、`900.00万`、`1000.00万`，以便读数自然。
15. 作为使用者，我希望较小的数值（不足一万）直接展示原始数字、不强加单位，以便小数不被异常折算。
16. 作为开发者，我希望 Token 单位换算被封装为一个通用工具函数并配单元测试，以便多处复用、口径一致、避免重复实现。
17. 作为使用者，我希望仪表盘"Token（今日）"卡片也带辅助单位小字，以便与统计页口径一致。
18. 作为使用者，我希望用量统计的总量卡片（输入/输出/缓存 Token）也带辅助单位小字，以便快速判断规模。
19. 作为使用者，我希望统计页"端点请求记录"（ranged 模式）能正常出数据、不再永远"加载中"，以便回溯某时间段的请求明细。
20. 作为使用者，我希望在同一时间筛选条件下，监控列表不会反复重新请求/闪烁，以便稳定浏览与翻页。
21. 作为使用者，我希望"历史记录"弹窗更大更宽，长表格有滚动区域，以便更舒适地翻阅历史聚合明细。

## Implementation Decisions

### R1 监控端点路由展示（后端 + 前端）
- **数据层**：在 `request_logs` 表新增两列 `inbound_path`、`upstream_path`（均可空文本），通过新增一条版本化迁移（**v5**）追加，沿用现有 `schema_version` 增量迁移机制（不破坏 v1–v4）。`request_logs` 仓储的插入与查询、`RequestLog` 模型结构体同步这两列。
- **采集点**：代理转发汇聚点（`RequestMeta`/`RequestRecord`）透传真实入站路径（`uri.path()`）与真实出站路径（转换时为 `/v1/chat/completions`，直通时为入站路径）。聚合器明细缓冲与事件 payload 一并带上两列。
- **前端展示**：监控表"入站/出站"列改为展示路径列；协议词（`inbound_format`）与上游主机（`upstream_url`）降级为辅助信息（如副标题/悬停）。旧行 `inbound_path/upstream_path` 为空时，按 `inbound_format` 推断兜底（claude→`/v1/messages`、openai→`/v1/chat/completions`）。
- **向后兼容**：新列可空、仅前向记录，不回填旧行；事件契约仅"增字段"。

### R2 用量同步异步化（后端 + 前端）
- 将 `sync_session_usage` 从同步 `#[tauri::command] fn` 改为 `async fn`，把"取连接 + `sync_all` 扫描解析"整体放入 `tauri::async_runtime::spawn_blocking`，避免阻塞主线程事件循环。连接池句柄（`DbPool`，r2d2 内部 `Arc`，可克隆）在进入阻塞闭包前从 `State` 取出。
- 解析与去重逻辑（`usage_local::sync_all` 及 claude/codex 读取器）**行为不变**，仅改变执行线程，结果一致。
- 前端 `UsagePanel` 进入面板触发的同步保持非阻塞（mutation 异步），加载态用现有 `sync.isPending` 表达；其余查询照常并行渲染，不被同步阻塞。

### R3 用量时间筛选（前端为主）
- `UsagePanel` 新增时间段筛选器（快捷项：今日/近 7 天/近 30 天/全部），**默认今日**，状态驱动三个查询的 `start/end`（`YYYY-MM-DD`）。
- 后端 `get_usage_summary/get_usage_by_model/get_usage_by_day` 已支持 `start/end/app_type`，**无需改后端**；前端服务层 `usageApi` 已透传 `start/end`，仅在面板补齐入参与 queryKey。
- 时间段计算复用 R4/R6 引入的稳定日期工具，避免 `Date.now()` 进 queryKey 的同类问题。

### R4 Token 单位工具函数（前端）
- 在 `src/lib/` 新增纯函数 `formatTokenCompact(n) -> string`（命名实现时定）：
  - 数值 ≥ 1e8 用"亿"，≥ 1e4 用"万"，否则原样 `toLocaleString()`、不加单位；
  - 折算保留两位小数，前缀"约等号"`≈`（如 `≈1.25亿`、`≈2.00万`）；
  - 对 0、负数、非有限值做防御（0 直接 `0`，负数取绝对值折算并保留符号或直接原样——实现时取简单稳健策略）。
- 该函数作为辅助单位文案来源，供统计页用量卡片、监控 Token、仪表盘 Token 等处复用；主数值仍展示精确值，单位文案作"辅助小字"。

### R5 仪表盘 Token 单位（前端）
- 仪表盘"Token（今日）"用 `StatCard` 既有 `hint` 槽位承载辅助小字（`formatTokenCompact(tokens)`），主值保留精确 `toLocaleString()`。

### R6 修复 ranged 监控一直加载（前端）
- 根因：`RequestMonitor` 的 `computeRange(rangeKey)` 在每次渲染都用 `Date.now()` 计算 `endMs`（`7d/30d` 的 `startMs` 同理），使 `["request-logs", …, startMs, endMs, …]` 这个 queryKey 每帧都变，React Query 视作新查询、永远 `pending`，UI 永久"加载中"，并持续打后端。`live` 模式 `range` 为空对象、key 稳定，故不受影响——与"仅统计页坏、仪表盘正常"现象一致。
- 修复：稳定化时间段计算——把范围按**天对齐**（`today` 取当日 0 点到当日 23:59:59.999 或次日 0 点的固定上界；`7d/30d` 以"今天"为锚按天对齐），并对计算结果按 `rangeKey`（必要时加"当天"标识）做 `useMemo`，保证同一筛选条件下 `startMs/endMs` 稳定。将范围计算抽为可单测的纯函数。

### R7 历史记录弹窗放大（前端）
- `HistoryDialog` 的 `DialogContent` 由 `max-w-3xl` 放大（如 `max-w-5xl`/`max-w-6xl`，实现时定），并给表格区域设定最大高度 + 纵向滚动，保证分页器与标题常驻、内容可滚动。

## Testing Decisions

测试只验证外部行为（输入→输出 / 落库结果 / 渲染结果），不绑定内部实现细节。优先复用既有最高层 seam，仅在必要处新增。

- **Token 单位工具函数（前端，新 seam，最高层）**：为 `formatTokenCompact` 写纯函数单元测试，覆盖量级边界（9999→不加单位、10000→`≈1.00万`、99,999,999→万档、100,000,000→`≈1.00亿`）、就近取档、两位小数、0、负数与非有限值防御。先例：`src/__tests__/filters.test.ts`、`i18n.test.ts` 的纯逻辑测试。
- **时间段计算稳定性（前端）**：把 ranged 范围计算抽为纯函数并单测——同一 `rangeKey`（与同一"当天"输入）多次调用返回相等的 `startMs/endMs`（防 queryKey 漂移回归），并校验各档边界（today/7d/30d/all）。先例：`filters.test.ts`。
- **监控路由展示（前端）**：扩展既有 `src/__tests__/RequestMonitor.test.tsx`，断言 `RequestLogTable` 在给定含 `inboundPath/upstreamPath` 的日志时，入站/出站列渲染路径；在缺省（空路径）时按 `inboundFormat` 渲染兜底路径。
- **用量时间筛选（前端）**：在 UsagePanel 相关测试中，断言默认筛选为"今日"并以对应 `start/end` 发起查询；切换筛选改变查询入参（通过 mock `usageApi` 校验调用参数）。
- **请求明细仓储（后端，既有 seam）**：扩展 `request_logs_repo` 测试，覆盖 `inbound_path/upstream_path` 的插入与回读（含空值），分页查询返回新列。
- **迁移（后端，既有 seam）**：在 `migration.rs` 测试中新增断言——v5 迁移后 `request_logs` 含 `inbound_path/upstream_path` 两列、且从 v4 旧库可平滑升级。先例：`v3_adds_cache_columns_and_request_logs`。
- **用量同步异步化（后端）**：行为不变，沿用 `usage_local` 既有解析/去重/取差 seam，确认现有测试仍通过即可（`spawn_blocking` 不改变解析结果，不新增断言）。

## Out of Scope

- 请求明细旧行的**路径回填**：仅前向记录真实路径，迁移前旧行靠协议推断兜底，不做历史回填。
- 用量统计的**成本/费用计算、定价表、趋势折线图**（延续第一轮 MVP 边界）。
- 用量数据的**文件监听 / 实时推送**（延续 MVP：进入 Tab 同步 + 手动刷新）。
- 用量同步的**细粒度进度条/百分比**：本轮仅保证异步不卡死 + 进行态提示，不做逐文件进度。
- Token 单位工具的**国际化/英文单位（K/M/B）**：本轮按中文"万/亿"，不做多语言单位切换。
- 监控/历史的导出、告警等增强能力。

## Further Notes

- 第一轮的 `get_stats` / `stats-updated` / `request-logged` 契约保持向后兼容，本轮 R1 仅"增列/增字段"，不改既有语义。
- R6 是纯前端回归修复，但同类陷阱（`Date.now()` 进 queryKey）在 R3 的时间筛选中也会出现，故 R3/R6 复用同一套稳定化日期工具，统一规避。
- R1 迁移 v5 与 R2/R3/R4/R5/R6/R7 相互解耦：R1 是唯一涉及后端迁移与采集点的工作；R2 是后端命令异步化；R3/R4/R5/R6/R7 为前端改造。建议构建顺序：R4 工具函数先行（被 R3/R5 复用）→ R6 监控修复 → R2 用量异步 → R3 时间筛选 → R1 路由路径（后端迁移+采集+前端列）→ R5 仪表盘单位 → R7 弹窗放大。
- "publish 到 issue tracker"在本项目落地为：本 PRD 文件（`状态：ready-for-agent`）+ `TASKS.md` 工作包 + `progress.csv` 进度行，沿用第一轮既有约定。
