# 2038 PRD: 页面自由布局、路由规则配置、日志四段详情与 Web 配置台

## 目标

在不破坏现有桌面端主流程的前提下，补齐以下四项能力：

1. 为每个菜单页增加统一的页面布局编辑能力，入口放在“切换为顶部导航”按钮旁边。
2. 新增独立菜单页管理路由、熔断、降级规则，并允许用户自定义保存。
3. 扩展日志能力，使日志菜单页能够展示单次请求的四段完整链路详情。
4. 在桌面端能力稳定后，提供 1:1 复刻现有界面的 Web 配置台。

## 范围

### 第一阶段：桌面端增强

- 每个菜单页支持布局编辑模式。
- 新增独立的规则配置菜单页。
- 后端补充请求链路详情采集与持久化。
- 日志页新增请求详情视图。

### 第二阶段：Web 配置台

- 抽象当前前端的数据访问层。
- 在 Rust 侧提供浏览器可访问的管理 API 和事件通道。
- 复用现有 React 页面，提供桌面端与浏览器端双宿主支持。

## 非目标

- 第一阶段不做任意像素级自由拖拽，不支持用户创建自定义新组件。
- 第一阶段不把所有 Tauri 特有能力搬到 Web 端；托盘、窗口控制等桌面专属能力在 Web 端可隐藏或替换。
- 不在第一阶段重写现有页面视觉设计。

## 用户故事

### Story 1：页面自由布局

作为桌面端用户，我希望每个菜单页都能进入布局编辑模式，以便按自己的使用习惯调整页面区块顺序、显隐和分栏方式。

#### 验收标准

- 所有菜单页都能从统一入口进入布局编辑模式。
- 布局编辑入口位于侧边导航底部“切换为顶部导航”按钮旁边。
- 每个页面的布局偏好独立存储，重启应用后保留。
- 用户可以恢复当前页默认布局。

### Story 2：独立规则配置菜单页

作为管理员，我希望在一个单独菜单页里管理路由、熔断和降级配置，而不是依赖写死的默认值。

#### 验收标准

- 新增单独菜单页承载全部规则配置。
- 路由、熔断、降级配置具备读取、编辑、保存、重置能力。
- 保存后后端立即加载新配置，必要时刷新运行时状态。
- 配置有默认值和基础校验，错误输入会被阻止保存。

### Story 3：日志四段请求详情

作为排障用户，我希望在日志页中看到每次请求的完整链路详情，以便快速判断问题发生在哪一段。

#### 验收标准

- 至少展示四个固定阶段：
  - 接收请求
  - 转发请求
  - 接收转发的请求
  - 响应请求
- 每段至少展示 URL、Method、Headers、Body、状态码或阶段结果。
- 敏感头和过长报文会被脱敏或截断。
- 可以从日志页定位到某次请求并查看详情。

### Story 4：Web 配置台

作为远程运维用户，我希望在浏览器中访问与桌面端一致的管理界面，而不是必须打开桌面应用。

#### 验收标准

- 浏览器端复用桌面端页面结构和主要交互。
- Web 端通过 HTTP API 和事件通道获取与修改数据。
- 同一套前端页面可根据宿主环境切换桌面适配器和 Web 适配器。
- 与桌面专属能力不兼容的功能在 Web 端有明确降级行为。

## 核心决策

### 决策 1：分两阶段落地

先完成桌面端的布局、规则页、日志详情，再做 Web 配置台。

原因：

- 当前前端直接依赖 Tauri `invoke/listen`，Web 化需要额外接口层。
- 日志四段详情和规则配置都需要先补齐后端数据与配置模型。
- 分阶段更利于按模块 scoped 提交与验证。

### 决策 2：布局能力采用“页面区块编排”而非“像素级画布”

第一阶段的自由布局仅支持对页面内既有区块进行：

- 顺序拖拽
- 显示或隐藏
- 预设分栏切换

原因：

- 更容易在现有 React 页面中落地。
- 更利于桌面端和 Web 端共享配置结构。
- 能满足大部分“自由调整页面布局”的实际诉求。

### 决策 3：规则配置从硬编码迁移到独立配置模型

当前熔断默认值、降级策略等逻辑主要位于 Rust 代码常量和分散逻辑中。后续将抽象为独立规则配置模型，由配置页维护。

### 决策 4：日志详情不复用普通日志流

请求四段详情属于结构化请求追踪数据，不应塞入普通日志行。它将扩展现有请求明细模型与持久化表结构，并在日志页中单独展示。

### 决策 5：Web 端通过适配器模式接入

前端会抽象统一的数据访问接口：

- Desktop adapter：继续使用 Tauri IPC
- Web adapter：改为 HTTP API + WebSocket 或 SSE

这样可以最大化复用现有页面和状态管理逻辑。

## 信息架构

### 导航

- 现有菜单保持不变。
- 新增一个规则配置页菜单项，名称为“规则配置”。
- 布局编辑入口位于导航底部控制区，与导航模式切换按钮并列。

### 页面布局模型

每个页面拥有自己的布局配置，至少包含：

- 页面标识
- 区块列表
- 区块顺序
- 区块显隐状态
- 容器布局模式

### 规则配置模型

规则配置由三部分组成：

- 路由规则
- 熔断规则
- 降级规则

要求：

- 后端有稳定默认值
- 前端按结构化表单展示
- 支持保存与恢复默认值

### 请求追踪模型

每条请求明细在现有字段基础上扩展四段追踪节点。每个节点至少包含：

- 阶段名
- URL
- Method
- Headers
- Body
- 状态码
- 时间戳或耗时

## 数据与存储约束

- 页面布局偏好持久化到前端本地 store。
- 规则配置持久化到后端配置存储，并支持启动恢复。
- 请求四段详情持久化到数据库，但需要长度限制、敏感字段脱敏和保留期清理。

## 错误处理

- 布局配置损坏时自动回退默认布局。
- 规则配置保存失败时保留表单状态并提示失败原因。
- 请求详情采集失败不能影响正常转发链路。
- Web 端不可用的桌面能力需要显式降级，而不是静默报错。

## 测试要求

### 桌面端阶段

- 前端组件和 store 的单测。
- Rust 配置模型、迁移、请求详情采集与查询单测。
- 页面交互的回归验证。

### Web 阶段

- 适配器层测试。
- HTTP API 与事件通道集成测试。
- 桌面适配器与 Web 适配器共用页面的回归验证。

## 交付顺序

1. 设计文档确认
2. 实施计划拆分
3. 桌面端布局能力
4. 规则配置页与后端配置模型
5. 日志四段详情链路
6. Web 配置台适配层与管理接口
7. 浏览器端接入与回归
